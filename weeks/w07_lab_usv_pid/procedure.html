Procedure

← Back to Lab 2 overview

  • The lab collects six step-response datasets per team: three parameter configurations (Baseline, Feedforward, Tuned) × two control channels (speed, yaw-rate).
  • The same field workflow runs three times; only the parameter set changes between cases.
  • Quality is checked at the field before each parameter-set change, so problems are caught while the boat is still in the water.

Plan for two field sessions. Getting clean step responses on the first day is rare. The first session typically surfaces issues (vehicle setup, parameter staging, pilot/scribe rhythm) that need fixing before clean data lands on a second day. Build in time between sessions for laptop-side processing.

Read this whole page (and the linked documentation) before the first field session. The procedure has a lot of moving parts, and the post-processing relies entirely on the scribe’s notes to navigate dozens of .mat files. Going in cold makes the field session much harder than it needs to be. Each team member should also have a clear mental model of the overall lab objectives and the three experiments before stepping onto the dock.

1. Pre-lab preparation

1.1 Day before

  • Collect: Boat, 900 MHz radio, R/C transmitter, ground laptop (with Mission Planner). Make sure they are in one place and ready to transport to the field.
  • Charge: Batteries for vehicle propulsion, autopilot, R/C transmitter, ground laptop.

1.2 Day of

  • Test card / data sheet printed — one row per dataset (six total) for parameter values, observations, and quality-check verdict. Template. This is just a template; students can make their own version if they want, but the key is to have a structured place to record the UTC timestamp and intent for each arm event, which is how the resulting .mat files get navigated during post-processing.
  • Parameter files staged on the laptop. You are given the baseline .param for the Base experiment.
  • Roles:
    • Planner — drives Mission Planner: monitors ARM and mode state, watches the real-time tuning graphs, and saves a timestamped parameter export at the start of each run.
    • Scribe — keeps the test card and records UTC time + intent at every arming event. The scribe’s notes are what makes the resulting tens of .mat files navigable during post-processing.
    • Pilot — drives the boat with the RC transmitter and executes the step inputs that exercise the closed-loop controllers.

2. Shore setup

As in Lab 1, confirm the following at the launch site before any data collection.

  • ⚠️ Safety. Stay clear of the propellers whenever the boat’s propulsion battery is connected.
  • SD card seated in the autopilot
  • Power on the autopilot.
  • Establish telemetry link with Mission Planner.
  • Load the baseline .param file and confirm the relevant ATC_* parameters are at the expected values. ATC = Attitude Control, see Ardu Common Abbreviations. See the Stabilization Layer reference for what each parameter does.
  • Power on the propulsion system.
  • Confirm ARM and that you can switch from MANUAL to ACRO modes.
  • You may need to re-bind the RC transmitter and/or calibrate sensors.

3. Experiments

The lab cycles through three experiments 1. Base: Ardu default PID gains.
2. FeedForward, adds feedforward from the Lab 1 System ID results. Tuned), applying the same procedure to each. Only the parameter set changes between cases (§3.4).

3.1 Per-arm rhythm and logging

The autopilot starts a new log file every time the pilot arms it. Treat each arm-disarm cycle as one experimental purpose: one cycle for the speed step trials, one for the yaw-rate trials. This keeps each .mat file scoped to a single channel, which makes the MATLAB time-series easier to inspect. Expect to accumulate dozens of .mat files over a session; the scribe’s notes (UTC time with seconds plus a brief intent description at every arm) are how you map each .mat back to what was being tested. The post-processed .mat filename includes the UTC timestamp.

3.2 Procedure for each experiment

Run the following sequence three times, once per experiment (Base → FeedForward → Tuned).

Set up the experiment

  1. Change the parameter set for this experiment via Mission Planner (Config → Full Parameter Tree). For Base, load the baseline .param. For FeedForward and Tuned, you will be changing the relevant ATC_* parameters as specified in the lab instructions.

  2. Verify the relevant ATC_* parameters at the vehicle. Mission Planner shows live values; see the Stabilization Layer reference for what each parameter does and which ones change between experiments.

Speed (throttle) step run — one arm-disarm cycle

  1. Pilot brings the USV to open water and to rest.
  2. Planner saves the current parameter file to the laptop with a timestamp in the filename (Mission Planner → Config → Full Parameter Tree → Save). Pilot then disarms and re-arms (closes the previous log, opens a fresh one). Scribe records UTC time + intent (e.g. “Base, surge step”).
  3. Pilot switches the autopilot into ACRO mode. Planner confirms ACRO in Mission Planner before the pilot proceeds.
  4. Repeat 3–5 times:
    • Apply a step throttle command at ~50% of full stick deflection (mid-range, keeping the actuator out of saturation while staying visible above noise).
    • Hold for 20–30 s so the surge response settles into a clear steady value.
    • Return throttle to zero; allow the boat to come back to rest before the next repeat.
  5. Pilot disarms to close the speed-trial log.

Yaw-rate (steering) step run — one arm-disarm cycle

  1. Pilot brings the boat to a low, approximately constant surge speed, typically around ~50% throttle so the rudder has flow but the boat isn’t fighting acceleration during the test. (At zero surge speed a steering command does nothing useful: the rudder needs forward water flow to produce a turning moment.)
  2. Planner saves the current parameter file with a timestamp. Pilot then disarms and re-arms. Scribe records UTC time + intent (e.g. “Base, yaw step”).
  3. Pilot switches the autopilot into ACRO mode. Planner confirms ACRO before the pilot proceeds.
  4. Repeat 3–5 times, alternating direction (positive then negative stick deflection so any asymmetry is visible):
    • Apply a step steering command at roughly ~50% rudder. Use the same step fraction in every experiment (Base, FeedForward, Tuned) so the metrics are directly comparable.
    • Hold until the yaw rate settles to a steady value.
    • Scribe records the surge speed at the time of each step, the step fraction, and the direction.
  5. Pilot disarms to close the yaw-trial log.

After the run

Note both log filenames and parameter values on the test card. The detailed quality-check on the captured data is done back in lab after the session — see §5 Post-processing and quality check. (Students with command-line Python comfort are welcome to run bin2mat.py and a quick MATLAB plot at the field, but the default is to defer the full check until the instructor has converted the day’s .BIN files in batch. Catching issues this way is part of why the lab usually needs two field sessions, not one.)

3.3 What changes between experiments

Experiment Parameter changes from previous Notes
Base Lab preset configuration; ArduRover stock PID gains; FF off; rate limiters off; throttle baseline off First case — fully load the lab baseline .param
FeedForward Set ATC_SPEED_FF and ATC_STR_RAT_FF to 1/K_dc from the Lab 1 plant models (see below). Same PID gains as Base.
Tuned Adjust ATC_SPEED_P/I/D and ATC_STR_RAT_P/I/D for best response. FF kept on (from FeedForward). Use Mission Planner’s real-time tuning graphs to iterate on candidate gains; only execute the full arm-disarm logging cycle (steps 3–12) for the gain values you decide to keep.

Setting the feedforward gains (FeedForward and Tuned experiments)

Set each channel’s feedforward gain to 1 / K_dc, where K_dc is the DC gain of that channel’s plant model from Lab 1:

  • Speed: G_speed(s) = 5.1 / (1.3 s + 1) → ATC_SPEED_FF = 1/5.1 ≈ 0.20
  • Yaw rate: G_yaw(s) = 0.82 / (0.51 s + 1) → ATC_STR_RAT_FF = 1/0.82 ≈ 1.22

3.4 Tuning the PID gains (Tuned experiment)

The Tuned experiment is iterative: try a candidate set of P / I / D gains, watch how the system responds in real time, adjust, repeat. Mission Planner is the primary instrument for this — its real-time tuning graphs let you see the response without going through the field-quality-check loop for every iteration.

The ArduRover docs page Tuning Speed and Throttle (the Desired Speed to Throttle PID Tuning section) is the canonical reference. The same general approach applies to the yaw-rate channel — there isn’t a parallel docs page for it, but the workflow is the same:

  1. Start from the FeedForward configuration. P, I, D at Base values; FF at 1/K_dc. The boat should already track the target reasonably well.
  2. Open Mission Planner’s real-time tuning graphs (Flight Data → Tuning checkbox). Plot the desired vs measured signal for the channel you’re tuning.
  3. One gain at a time, small steps. Increase P first (faster response, more overshoot). If overshoot is excessive, back off P or add a small amount of D. If the steady-state error is non-zero after the FF contribution, increase I.
  4. Don’t log every iteration. Watching the real-time graphs is enough during the iteration. Only when you’ve settled on a candidate gain set worth keeping, run the full arm-disarm logging cycle (steps 3–12 of §3.2) so you have a clean .mat file with the parameter snapshot for the post-processing.
  5. Stop when “good enough.” A perfect tune is rare in one field session. Aim for a noticeable improvement in rise time and settling vs the FeedForward case, with overshoot still controlled.

3.5 Common gotchas

  • Don’t change parameters while ARMed. Parameter changes get baked into the active log in a way that makes it hard to verify what was active when. Disarm, change, save the new param file, then re-arm.
  • Verify ACRO mode every time. A test in MANUAL looks identical to one in ACRO from the pilot’s perspective, but the closed-loop controllers aren’t engaged. The planner’s ACRO confirmation in step 5 / step 10 is the safety net; don’t skip it.
  • Inter-step gap too short. Between repeats within a run, allow the response to fully settle (return to rest, or to baseline yaw rate) before the next step. A short gap mixes one step’s tail into the next step’s start.
  • Same step fraction across Base / FeedForward / Tuned. Comparing rise time, overshoot, and steady-state error across experiments should be done using equivalent amplitude step inputs for the tests. Since the pilot is setting the amplitue, try to be consistent with command amplitude across the three test cases.
  • One channel per arm-disarm cycle. Don’t intermingle speed and yaw steps in the same log; it makes the MATLAB inspection harder than it needs to be.
  • Two timezones. ArduPilot logs and the post-processed .mat filenames are in UTC. Scribe notes are in local time. The post-processing step adds the UTC stamp to the filename automatically; the scribe should record UTC at the field too so the two match without arithmetic later. Instructors use the UTC format YYYYMMDD_HHMMSS for the appended timestamp.

4. End-of-day

Wrap up the field session.

  • Upload raw .BIN logs to the shared data repository (ME2801_USV_Shared). Add a subfolder for your team (e.g. TeamX) and put everything there:
    • the raw .BIN files (instructors will convert to .mat);
    • the scribe’s notes / test card;
    • the timestamped .param exports from Mission Planner;
    • any photos or other field-session materials.
  • Disconnect all batteries.
  • Drain the boat, put batteries on chargers, and stow the vehicle with the rest of the team’s lab materials (boat, 900 MHz radio, R/C transmitter).

5. Post-processing and quality check

The quality-check step can’t happen until the .BIN logs have been converted to .mat, which is run from the command line.

  1. Convert the day’s .BIN files to .mat. Work with the instructors to run bin2mat.py on the uploaded files. The resulting .mat files are what the MATLAB analysis scripts expect, and the UTC timestamp in each filename matches the scribe’s notes.
  2. Visual inspection per file. Adapt closedloop_assess_example.m to your own log filenames and run it once per .mat file. The script produces the timeseries and PID-internals figures and computes step-response metrics. Suggested order is to run it across all six experiment-channel .mat files first, just to confirm each captured a clean step response.
  3. For each step in the figures, verify:
    • Setpoint reaches the commanded value (no rate-limited ramp instead of a step).
    • Measured response settles to a steady value within ~10 s.
    • Control effort is not pinned at ±1 for an extended period (saturation).
    • PIDA / PIDS log channels are present (PID-internal logging is on; PIDA for speed, PIDS for yaw-rate — the naming is counterintuitive).
    • No long gaps in the GPS or STER sample stream.
  4. If a check fails for a dataset: that channel/experiment needs to be re-collected on the next field session. Update the test card and flag it for the next session’s plan.
  5. Isolating the step window and reading off the quantitative metrics is done together in class next week — the initial plotting and visual inspection above can be done by each team on their own.
