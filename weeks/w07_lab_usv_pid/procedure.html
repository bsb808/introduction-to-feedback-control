Procedure

← Back to Lab 2 overview

Working draft. Procedure is being refined as the lab runs this quarter; expect updates day-to-day.

The lab collects six step-response datasets — three parameter configurations (Baseline, Feedforward, Tuned) × two control channels (speed, yaw-rate)% CLAUDE: Change to a bullet point list. The same field workflow runs three times; only the parameter set changes between cases. Quality is checked at the field before each parameter-set move, so problems are caught while the boat is still in the water.

1. Pre-lab preparation

% CLAUDE: We need two things here. Day before and day-of prep. The former is to make sure you have your boat, 900 MHz radio, and R/C transmitter and ground laptop ready to go.
- Team roles — planner (Mission Planner operator) scribe and pilot. - Ground laptop ready — Mission Planner installed and connected to the radio. - Parameter files staged — three .param files corresponding to experiments A / B / C, located under utils/ardu_params/. % CLAUDE: They are only given the baseline .param file; they have to edit it to create the other two. - Vehicle ready — battery charged, props/rudder clear, RC bound, GPS lock confirmed. - Test card / data sheet — one row per dataset (six total) for noting parameter values, observations, and quality-check verdict.

2. Field setup

Steps to confirm at the launch site, before any data collection.

  • Launch and recover the USV safely; clear of swimmers and obstructions.
  • Establish telemetry link (Mission Planner connected at expected baud).
  • Confirm flight mode is ACRO.
  • Confirm logging is enabled and a new log file starts on each arm (LOG_FILE_DSRMROT = 1).
  • Trim and zero check on RC sticks (no commanded motion at center stick).

3. Experiment execution

For each of the three experiments (A → B → C), repeat the same field workflow:

  1. Load the parameter set for this experiment via Mission Planner (Config → Full Parameter Tree → Load from file).
  2. Verify the relevant ATC_* parameters at the vehicle (Mission Planner shows live values).
  3. Speed step responses — see §4.1 for stick technique.
  4. Yaw-rate step responses — see §4.2.
  5. Field quality check — pull the log, plot the candidate steps, decide keep / repeat (§5).
  6. Save and label — note the log filename and parameter values on the test card.

5. Quality check (before leaving the field)

Run the visual inspection on the laptop while the boat is still launched, so any failed dataset can be recollected on the spot.

  1. Pull the latest .BIN log from the autopilot SD card or via Mission Planner.
  2. Run bin2mat.py to convert: produces a .mat file alongside the .BIN.
  3. Run proc_closedloop_response.m in MATLAB to plot the timeseries.
  4. Check each step in the figure:
    • Setpoint reaches the commanded value (no rate-limited ramp instead of a step)
    • Measured response settles to a steady value within ~10 s
    • Control effort is not pinned at ±1 for an extended period (saturation)
    • PIDT / PIDS log channels are present (PID-internal logging is on)
    • No long gaps in the GPS or STER sample stream

If any check fails for the dataset just collected → recollect before moving on. If it’s a parameter or vehicle issue → resolve, then continue.

6. End-of-day

Wrap up the field session.

  • All six .mat files transferred to the team’s shared location.
  • Test card filled in (one row per dataset, with quality-check verdict).
  • Vehicle returned to nominal parameter configuration (rate limiters restored, FF restored, throttle baseline restored) — load a .param file with the post-lab nominal values.
  • Battery on charger, vehicle stowed.

7. Hand-off to next week

Datasets and notes are the input to the Week-8 analysis: process each .mat with closedloop_stepinfo.m to extract rise time, settling time, overshoot, and steady-state error, then compare across the three experiments.
